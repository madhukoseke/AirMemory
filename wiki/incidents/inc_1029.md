# INC-1029 — Row count mismatch

Status: RESOLVED  
Incident ID: `INC-1029`  
DAG: `customer_daily_migration_dag`  
Task: `validate_row_counts`  
Failure Category: `row_count_mismatch`  
Owner: customer-data-platform  
Confidence: 0.93  

## Summary

`validate_row_counts` failed because HANA source count (1588) did not match BigQuery target count (1297). Downstream Looker `customer_metrics` looked stale; lineage pointed back to this upstream mismatch.

## What Happened

```text
ROW_COUNT_MISMATCH: HANA 1588 / BigQuery 1297 / diff 291
```

Late-arriving HANA records carried offset `processing_date` values and were excluded by an exact-date filter.

## Likely Root Cause

Validation used exact `processing_date = system_date`, which missed late-arriving records in `hana.customer_master` before `bq.prod.customer_master` was complete.

## Accepted Fix

Widen the validation window to `system_date - 3` through `system_date + 3`, then rerun only `validate_row_counts` and `dq_reconciliation_check`.

Resolution: `res-window-3-day`

## Rejected Fixes / What Not To Repeat

- Full DAG clear / reload (`res-full-dag-clear`) — unsafe on incremental downstream tables

## Recommended Next Steps

1. Apply ±3 day processing_date window
2. Rerun validate_row_counts only
3. Confirm Looker customer_metrics freshness
