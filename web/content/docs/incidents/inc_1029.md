# INC-1029 — Row count mismatch

Status: RESOLVED  
Incident ID: `INC-1029`  
DAG: `customer_daily_migration_dag`  
Task: `validate_row_counts`  
Failure category: `row_count_mismatch`  
Owner: customer-data-platform  
Confidence: 0.93  

## Summary

`validate_row_counts` failed because HANA source count (1588) did not match BigQuery target count (1297). Downstream `publish_metrics` and Looker `customer_metrics` looked stale even though the paged task was upstream validation.

## What Happened

- Source: `hana.customer_master` — 1588 rows
- Target: `bq.prod.customer_master` — 1297 rows
- Diff: 291
- Error: `ROW_COUNT_MISMATCH`

Late-arriving HANA records carried offset `processing_date` values and were excluded by an exact-date filter.

## Root Cause

Validation used exact `processing_date = system_date`. Late-arriving records in `hana.customer_master` were excluded before `bq.prod.customer_master` was complete.

## Accepted Fix

Widen the validation window to `system_date - 3` through `system_date + 3`, exclude rows already matched in BigQuery, then rerun **only**:

1. `validate_row_counts`
2. `dq_reconciliation_check`

Do not run `publish_metrics` until validation passes.

Resolution ID: `res-window-3-day`

## Rejected Fixes

- **Full DAG clear / reload** (`res-full-dag-clear`) — unsafe. Downstream tables are incremental; a full rerun can duplicate `bq.prod.customer_metrics`.

## Lineage

```text
hana.customer_master
  → bq.prod.customer_master
    → bq.prod.customer_metrics  (Looker)
```

A symptom on Looker `customer_metrics` can still resolve to this upstream incident via `DOWNSTREAM_OF` edges.

## Prevention

- Prefer windowed date validation over exact-date equality for late sources
- Gate `publish_metrics` on validation success
- Keep rejected full-clear workarounds out of recall (`forget`)
